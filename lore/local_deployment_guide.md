# Local EVM Deployment and LAU Verification Guide

This guide details the deployment of the entire **Dysnomia** Solidity system, the creation of a user **LAU** contract, and verification of user metrics (Identity, Soul, and Aura) on the local persistent EVM (Anvil).

---

## 1. Local Persistent EVM Configuration
The EVM (Anvil) runs persistently on port `8545` and serializes state changes directly to a file to retain state across restarts:
```bash
anvil --port 8545 --state state.json
```

All address outputs are securely saved inside [user_config.json](file:///home/mariarahel/src/tsfi2/atropa_pulsechain/config/user_config.json) under the `localhost` network key.

---

## 2. Compiling and Deploying the Dysnomia System

### Compilation
Solidty compilation targets EVM version `shanghai` with optimization enabled:
```bash
bash compile_linux.sh
```

### Full Deployment
A dedicated script, [deploy_dysnomia_system.js](file:///home/mariarahel/src/tsfi2/atropa_pulsechain/scripts/deploy_dysnomia_system.js), deploys the full network sequence including domains:

1. **System Core**: `VMREQ` $\rightarrow$ `SHAFactory` $\rightarrow$ `SHIOFactory` $\rightarrow$ `YI` $\rightarrow$ `ZHENG` $\rightarrow$ `ZHOU` $\rightarrow$ `YAU` $\rightarrow$ `YANG` $\rightarrow$ `SIU` $\rightarrow$ `VOID` $\rightarrow$ `ATTRIBUTE` $\rightarrow$ `LAUFactory` $\rightarrow$ `STRINGLIB`
2. **Mock Hecke Mapping**: Deploys `MockHecke` to resolve compliments and meridians for mapping.
3. **Reactions and Directory Entry**: Deploys `COREREACTIONSLIB` $\rightarrow$ `CHO` $\rightarrow$ registers `CHO` for talking in the reaction core.
4. **Map Grid**: Deploys the main `MAP` contract.
5. **QING Subspaces**: Automatically initializes the `VOID_QING` workspace via the map grid (`MAP.New()`), then deploys the sequential subdomains (`QI` $\rightarrow$ `MAI` $\rightarrow$ `XIA` $\rightarrow$ `XIE` $\rightarrow$ `CHAN` $\rightarrow$ `SEI` $\rightarrow$ `CHOA` $\rightarrow$ `CHEON`).

To execute this sequence:
```bash
node scripts/deploy_dysnomia_system.js
```

## 3. Creating the `mariarahel` User LAU
User identities are represented by `LAU` contracts generated by the factory. To create the contract and claim the identity:

1. Run [create_mariarahel_lau.js](file:///home/mariarahel/src/tsfi2/atropa_pulsechain/scripts/create_mariarahel_lau.js):
   ```bash
   node scripts/create_mariarahel_lau.js
   ```
2. This invokes `LAUFactory.New("mariarahel", "MARIARAHEL")`.
3. To secure ownership, it immediately executes `LAU.Username("mariarahel")` to write the attribute to the decentralized attribute storage.
4. The output user address is stored in config as `"mariarahelLau"`.

## 4. Verifying LAU Functions
We validated the contract features via [test_mariarahel_lau.js](file:///home/mariarahel/src/tsfi2/atropa_pulsechain/scripts/test_mariarahel_lau.js):
```bash
node scripts/test_mariarahel_lau.js
```
The test suite ensures correct execution of:
* Constant reads (`Type()`)
* Attribute storage writes (`Username(string)`)
* Logging via Chat logs (`Chat(string)`)
* Alias assignments (`Alias(address, string)`)
* Leaving current playing space (`Leave()`)
* State resets (`Void(bool, bool)`)

## 5. Retrieving Identity, Soul, and Aura
The `Saat` array containing the user's primary characteristics can be queried:
* `Saat(0)` $\rightarrow$ Identity (Pole)
* `Saat(1)` $\rightarrow$ Soul
* `Saat(2)` $\rightarrow$ Aura

Run the verification tool:
```bash
node scripts/get_lau_metrics.js
```
